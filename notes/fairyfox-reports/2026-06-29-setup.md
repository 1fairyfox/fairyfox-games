# Process report — new-project setup (2026-06-29)

**Procedure:** `new-project-setup` (bootstrapping fairyfox-games into the mesh).
**Outcome:** repo created, scaffolded, Ink Bloom moved in, registered hub-side,
released v0.1.0. Themed docs site deferred (landing page first pass only).

## What was done

- Created the public repo `junebug12851/fairyfox-games` with `dev`/`main`.
- Scaffolded a **monorepo** (`games/<slug>/`) rather than the runbook's implicit
  single-app shape — the project is a *collection*, so each game is a self-contained
  subfolder with its own core/test/shell/README.
- Adopted mesh-aware `CLAUDE.md` (with the fairyfox-system standing instruction), the
  notes skeleton, `VERSION`, `.gitignore`.
- Moved in **Ink Bloom**, already engineered to standard (pure core + 20 tests).
- Registered in both registries (`hub/registry.yml`, `_data/projects.yml`) as an
  `integrated` project pinned beside the site card.

## What was rough / didn't fit a fresh monorepo

- **The runbook assumes one app per repo.** "Set the starting version", "build the
  project", "the themed docs site" all read as singular. For a *collection* monorepo
  the unit of quality is the per-game folder, and the repo-level docs site is really a
  landing/index over many games. The runbook could note the monorepo/collection shape
  as a recognized variant.
- **Docs-site step is heavy for a static games repo.** The games ARE the site (static
  Pages); a separate "themed docs site" overlaps with the landing page. We satisfied
  the hard requirement (the "← Back to Fairy Fox" way-home link) on the landing page
  and deferred a fuller themed pass, marking `adopts_hub: false` honestly until then.
- **Integrated projects need a render tier.** The site sorted projects by date with
  the meta hub last; an `integrated` project that must pin beside the hub needed a new
  middle tier in `index.html`/`projects.md`. Worth capturing as a pattern if more
  site-integrated nodes appear.

## Suggestions for the runbook

- Add a short "collection / monorepo" note: version + tests + docs apply per-game;
  the repo-level site is a landing index.
- Clarify that for a static project the Pages site can double as the docs site, so
  long as the way-home link and theme are present.
